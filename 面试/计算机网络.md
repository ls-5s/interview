# HTTP 几个版本的区别

- HTTP/0.9 - 单行协议
只有 GET 请求行，无请求头和请求体
只能传输 HTML 文件，以 ASCII 字符流返回
无响应头
- HTTP/1.0 - 多类型支持
支持多种文件类型传输，不限于 ASCII 编码
引入请求头和响应头( key-value 形式)
每个请求都需要建立新的 TCP 连接
- HTTP/1.1 - 持久连接
引入持久连接( keep-alive )：一个 TCP 连接可传输多个 HTTP 请求
默认开启 keep-alive，通常限制 6-8 个并发连接
存在队头阻塞问题：前面的请求阻塞会影响后续请求
引入 Host 字段，支持虚拟主机
引入 Chunk transfer 机制处理动态内容长度
- HTTP/2 - 多路复用
一个域名只使用一个 TCP 长连接
引入二进制分帧层，实现多路复用
可对请求设置优先级
引入 HTTPS(HTTP + TLS) 加密

# HTTP 常见的状态码

200 请求成功，请求在服务端被正确处理
204 响应成功，没有数据
205 服务器处理成功，浏览器应重置文档视图
206 服务器成功处理了部分get请求
301 资源永久重定向
302 资源临时重定向
303 让你查看其他地址
304 请求的资源没有修改，服务端不会返回任何资源，协商缓存
400 请求语法错误，服务器看不懂
401 请求没有携带信息，比如 token 认证失败
403 请求被拒绝、敏感词
404 找不到资源
500 服务器内部错误，无法完成请求
501 服务器不支持当前请求所需的功能
503 服务器系统维护或者超载，暂时无法处理客户端的请求

# GET 和 POST 请求的区别

GET：查数据、URL 传参、可见、缓存、幂等、大小有限
POST：改数据、body 传参、安全、不缓存、不幂等、大小无限制

# 什么是跨域？如何实现跨域通讯？

跨域 = 浏览器的同源策略限制不同源之间的资源访问；跨域通讯 = 绕过或利用规则实现不同源的数据交互。
📌 同源策略（Same-Origin Policy）

浏览器的安全机制，要求三者必须一致：

协议（http / https）

域名（example.com）

端口（80 / 3000）
✅ 【2. 为什么会有跨域（Why）】

本质是为了防止：

XSS 攻击

CSRF 攻击

恶意网站读取用户敏感信息（cookie / token）

👉 浏览器主动限制，而不是服务器限制

📌 原理

服务器加响应头，告诉浏览器“我允许你跨域”

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type
📌 分两种请求
✔ 简单请求

直接发

✔ 复杂请求（会预检 OPTIONS）

实时聊天

# HTTP 请求跨域时为何要发送 options 请求

发送 OPTIONS 请求是浏览器的“预检机制（Preflight）”，用于在真正请求前确认服务器是否允许这次跨域操作。

OPTIONS 请求是 CORS 的预检机制，用于在非简单请求前由浏览器自动发送，询问服务器是否允许该跨域请求。
预检请求会携带 Access-Control-Request-Method 和 Access-Control-Request-Headers，说明即将发送的请求方式和头信息。
服务器通过 Access-Control-Allow-Methods 和 Access-Control-Allow-Headers 进行响应，浏览器校验通过后才会发送正式请求。
其核心目的是保障安全性并避免无效请求，同时可以通过 Access-Control-Max-Age 缓存预检结果优化性能。

# 如何理解 cookie

它是浏览器的一小段数据，每次请求会自动带给服务器，服务器会根据这个数据判断用户
它的执行过程
：服务器生成，浏览器储存
由服务器生成，浏览器存储
浏览器第一次请求服务器时，服务器在响应头里通过 Set-Cookie 字段把 Cookie 发给浏览器。
浏览器收到后，会自动把 Cookie 保存到本地（硬盘或内存）。

每次请求自动携带对应域名下的 Cookie
之后浏览器再次向这个域名发请求时，会自动在请求头 Cookie 里带上之前存的内容。
遵循同源策略：哪个域名下发的，就只带给哪个域名，不会乱传。

. 可设置过期时间
服务器设置 Max-Age 或 Expires，Cookie 就变成持久 Cookie。
到了指定时间，浏览器会自动删除这个 Cookie。

. 默认是会话 Cookie，浏览器关闭就删除
不设置过期时间时，就是会话 Cookie，存在内存里。
浏览器窗口一关，内存释放，Cookie 直接消失，下次打开就没了。

般是：

登录态（sessionId / token）
用户偏好（语言、主题）
跟踪信息（埋点、统计）

Cookie 为什么不安全？

 1. 会被 XSS 跨站脚本窃取
黑客在你的网页注入恶意 JS 代码
JS 可以直接通过 document.cookie 拿到所有 Cookie
黑客把 Cookie 发到自己服务器，就等于偷走了你的登录身份
后果：黑客直接用你的 Cookie 登录你的账号
 2. 会被 CSRF 跨站请求伪造
Cookie 最大特性：同域名请求自动携带
黑客做一个恶意页面 / 链接，诱导你点一下
恶意页面悄悄向「银行 / 后台」发请求（转账、删数据）
浏览器会自动带上你已登录的 Cookie
服务器以为是你本人操作，直接执行操作
这就是 Cookie 最致命的两个安全问题。

 3. HttpOnly → 彻底防 XSS 窃取 Cookie
作用：
服务器在 Set-Cookie 里加上 HttpOnly 属性
浏览器禁止 JS 读取这个 Cookie（document.cookie 拿不到）
就算页面有 XSS 漏洞，黑客也偷不走 Cookie
Demo 里的写法：
js
res.setHeader('Set-Cookie', 'username=zhangsan; HttpOnly')

. SameSite → 从源头防 CSRF
作用：限制 Cookie 只能在本网站携带，跨站请求不自动带，直接掐断 CSRF 的核心条件。
三个常用值：
Strict：最严格，完全禁止跨站携带
Lax：常用默认，只允许安全的 GET 跳转携带，POST / 表单 / 异步请求跨站都不带
None：必须配合 HTTPS，允许跨站（第三方登录 / 支付才用）
全站用 HTTPS 加密传输，中间人看不到明文
给 Cookie 加 Secure 属性：只在 HTTPS 下发送，HTTP 绝不发送

# 为何现代浏览器都禁用第三方 cookie

🧠 背景理解
1️⃣ 什么是第三方 Cookie？
第一方 Cookie：用户访问的网站自己设置的 Cookie
例：bank.com 登录状态 Cookie
第三方 Cookie：非当前访问域名设置的 Cookie
例：用户访问 news.com，页面里嵌入广告 ad.com，ad.com 设置的 Cookie 就是第三方 Cookie

2️⃣ 为什么问题大？
跨站跟踪：
广告商可以在不同网站嵌入 ad.com 的内容
浏览器自动携带 Cookie → 可以追踪用户跨站浏览行为
隐私泄露：
用户浏览习惯、兴趣、账号信息被偷偷收集
安全问题（延伸 CSRF）：
第三方 Cookie 可能被滥用做跨站攻击

现代浏览器禁用第三方 Cookie，主要原因是保护用户隐私和防止跨站跟踪。第三方 Cookie 可以在不同网站间追踪用户行为，容易造成隐私泄露和安全风险（如 CSRF / 恶意广告）。因此浏览器限制第三方 Cookie，用户浏览更安全，隐私更受保护。
默认阻止第三方 Cookie

# 什么是 JWT 描述它的工作过程

JWT（JSON Web Token）是一种紧凑、URL安全的自包含令牌，用于在客户端和服务器之间安全传递用户身份信息。

🧩 JWT 的结构

JWT 由三部分组成（中间用 . 分隔）：

header.payload.signature

Header（头部）

描述令牌类型和签名算法
{
  "alg": "HS256",
  "typ": "JWT"
}
Payload（有效载荷）
存放数据（声明 / Claims），常见有：
sub：用户 ID
iat：签发时间
exp：过期时间
自定义信息（如角色、权限）
Signature（签名）
用 Header + Payload + 密钥生成，用于验证数据完整性

公式：

HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(paylo

🏃 工作过程（大厂面试喜欢流程图思路）

假设用户登录系统：

1️⃣ 用户登录
用户提交用户名/密码 → 后端验证

后端生成 JWT：

{
  "header": { "alg": "HS256","typ":"JWT" },
  "payload": { "sub": "123", "role":"admin","exp": 1680000000 },
  "signature": "abc123..."
}
返回给前端（通常在响应体里）
2️⃣ 客户端保存
前端存到：
localStorage 或 sessionStorage 或 Cookie
后续请求附带 JWT（通常放在请求头 Authorization: Bearer <token>）
3️⃣ 服务端验证
服务端收到请求 → 提取 JWT → 校验签名
校验通过：
说明数据没被篡改
提取用户信息（ID、权限）
校验失败：
拒绝请求（401 / 403）

# JWT 如何自动更新 token ？

JWT 自身是 有过期时间的无状态 token，一旦过期就不能用。为了用户体验，不用每次登录都重新输入密码，需要刷新 token（Refresh Token）机制来“自动更新”。
// 前端示例代码
async function request(url, options) {
  try {
    const res = await fetch(url, {
      ...options,
      headers: {
        Authorization: `Bearer ${getAccessToken()}`,
      },
    })

    if (res.status === 401) {
      // access token 过期，尝试刷新
      const newToken = await refreshToken()
      if (newToken) {
        // 使用新 token 重试请求
        return request(url, options)
      } else {
        // refresh token 也过期，跳转登录
        redirectToLogin()
      }
    }

    return res
  } catch (error) {
    console.error(error)
  }
}

# 什么是 SSO 单点登录，描述它的工作过程

SSO（Single Sign-On，单点登录）是一种认证机制，用户只需登录一次，就可以访问多个相互信任的系统，而不需要每个系统重复登录。
🧩 SSO 的工作流程（最常见：基于 Cookie / Token / JWT）

假设有三个系统：应用 A、应用 B、认证中心 Auth

1️⃣ 用户访问受保护系统
用户访问 应用 A（未登录）
系统发现没有登录 → 跳转到 认证中心（Auth） 登录页面
2️⃣ 用户登录认证
用户在认证中心输入账号密码 → 认证通过
认证中心生成 统一的认证票据（Token / Cookie / JWT）
返回用户浏览器，并记录登录状态
3️⃣ 访问应用系统
用户浏览器携带认证票据访问 应用 A
应用 A 验证票据 → 登录成功
无需再输入账号密码
4️⃣ 访问其他系统
用户访问 应用 B
浏览器携带已有票据 → 应用 B 直接验证
登录成功 → 无需重复登录

整个 SSO 完整流程（超清晰版）
场景
认证中心：端口 3000（管 “你有没有登录”）
应用 A：端口 4000
应用 B：端口 5000
第 1 步：用户第一次访问「应用 A」
你打开浏览器访问：<http://localhost:4000>
应用 A 做的事：
去认证中心问：“这个人有 SSO 登录票吗？”
认证中心一看：浏览器里没有 sso_ticket Cookie
结果：没登录 → 应用 A 把你跳去认证中心登录页
第 2 步：在认证中心登录，生成 SSO 登录票
你输入账号：admin，密码：123456，提交登录
认证中心做的事：
验证账号密码正确
生成一串随机字符串 = SSO 登录票
javascript
运行
const ssoTicket = 'sso_xxxxxxxx'
把这张票存到自己的内存（SSO_SESSIONS）
把票写入你的浏览器 Cookie
结果：你现在全局登录了，浏览器里多了一个 sso_ticket
第 3 步：回到应用 A，自动登录成功
认证中心登录完，把你跳回应用 A
应用 A 再次做判断：
拿着你浏览器里的 sso_ticket 去问认证中心
认证中心查自己的记录：这张票有效，用户是 admin
结果：应用 A 直接让你进去，不用再登录
第 4 步：你再去访问「应用 B」
你打开：<http://localhost:5000>
应用 B 做的事：
同样去认证中心校验你浏览器里的 sso_ticket
认证中心：票有效，已登录
结果：应用 B 直接让你进去，完全不用输账号密码

你输账号密码 → 认证中心生成 SSO 登录票（存在 Cookie）
→ 作用：证明你已经全局登录
应用 A/B 校验 SSO 票 → 确认你已登录
✅ 关键一步：生成对应 Token
应用自己生成一个 专属业务 Token（就是你之前代码里的短 Token）
→ 作用：负责权限判断、接口访问
后续所有接口请求，都带这个 Token，不带 SSO 票

# 什么是跨域？如何实现跨域通讯？

浏览器有同源策略（Same-Origin Policy，SOP）：
保护用户数据，防止恶意网页读取别的站点数据
同源 = 协议 + 域名 + 端口 都相同
跨域 = 任意一项不同
💡 举例：
页面：<https://www.a.com>
请求接口：<https://api.a.com> → 跨域（端口/子域不同）
请求接口：<http://www.a.com> → 跨域（协议不同）

🔧 常见跨域通信方案（面试重点）
1️⃣ CORS（最主流✅）
Cross-Origin Resource Sharing
原理：
浏览器发请求前先发 OPTIONS 预检请求（部分请求）

服务器在响应头加：

```js
// 🔥 CORS核心配置（完全匹配你要的3个响应头）
app.use(cors({
  // 允许的前端域名（固定写你要的 https://www.a.com）
  origin: 'https://www.a.com',
  // 允许的请求方法
  methods: ['GET', 'POST'],
  // 允许的请求头
  allowedHeaders: ['Content-Type'],
  // 允许携带Cookie（可选，需要就开）
  credentials: true
}));
```

Access-Control-Allow-Origin: <https://www.a.com>
Access-Control-Allow-Methods: GET,POST
Access-Control-Allow-Headers: Content-Type
浏览器检查是否允许，如果允许才发真实请求
✅ 优点：官方标准、灵活、支持复杂请求

```js
// 导入 Vite 的配置定义函数（用于类型提示和配置校验）
import { defineConfig } from 'vite'
// 导入 Vite 官方 Vue3 插件，让 Vite 支持解析和编译 Vue 文件
import vue from '@vitejs/plugin-vue'

// 导出 Vite 配置对象（defineConfig 用于优化配置提示）
export default defineConfig({
  // 插件配置：注册 Vue 插件，启用 Vue 支持
  plugins: [vue()],
  // 开发服务器配置（本地运行 npm run dev 时的服务配置）
  server: {
    // 👇 前端开发环境【跨域代理】核心配置
    // 作用：解决前端调用后端接口时的浏览器跨域问题
    proxy: {
      // 匹配规则：所有 以 /api 开头的前端请求，都会走这个代理
      '/api': {
        // 代理目标：后端服务的真实地址（请求最终会转发到这里）
        target: '<http://localhost:3000>',
        // 🔥 核心配置：开启跨域
        // 原理：修改请求头中的 Origin，伪装成同源请求，解决跨域报错
        changeOrigin: true,
        // 路径重写：修改请求的 URL 路径
        pathRewrite: {
          // 正则匹配：把请求路径开头的 /api 替换为空字符串
          // 例：前端请求 /api/user/login → 转发后变成 /user/login
          '^/api': ''
        }
      }
    }
  }
})
```

后端地址：<http://localhost:3000>
前端写接口时统一加 /api 前缀，代理会自动转发到后端 3000 端口。

# HTTP 请求跨域时为何要发送 options 请求

它是来做预检请求的，来确定跨域请求是否安全，是否被服务器允许

为什么会有跨域问题？
浏览器同源策略：同源策略限制了脚本从远程 URL 加载资源，比如图片、脚本、样式表等等。

、为什么需要 OPTIONS（预检请求）？

不是所有跨域请求都会发 OPTIONS，只有**“复杂请求”**才会。

👉 浏览器的思路是：

“这个请求看起来有点危险，我先问一下服务器能不能发。”

于是就有了：

👉 预检请求（Preflight Request） = OPTIONS 请求

三、什么时候会触发 OPTIONS？
满足以下任意条件，就会触发：

1️⃣ 请求方法不是简单方法
❌ PUT
❌ DELETE
❌ PATCH

2️⃣ 请求头不“简单”

比如你常写的：

headers: {
  "Content-Type": "application/json",
  "Authorization": "Bearer xxx"
}

👉 就会触发预检！

3️⃣ Content-Type 不是简单类型

只有这三种是“简单”：

text/plain
multipart/form-data
application/x-www-form-urlencoded

👉 application/json ❌ → 会触发 OPTIONS

四、OPTIONS 请求到底做了什么？

浏览器会先发一个请求：

OPTIONS /api/user HTTP/1.1
Origin: <http://localhost:3000>
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization

👉 意思是：

“我要用 POST + 带这些头，请问你允许吗？”

完整预检流程（浏览器自动干的）

1. 浏览器发现：这是跨域 + 非简单请求
只要满足你前面说的任意一条：
方法是 PUT/DELETE/PATCH
头带了 Authorization 等自定义头
Content-Type 是 application/json
浏览器就知道：不能直接发真实请求，必须先问服务器。
2. 浏览器偷偷发 OPTIONS 预检请求
plaintext
OPTIONS /api/user HTTP/1.1
Origin: <http://localhost:3000>
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type,Authorization
翻译成人话：
服务器大哥，我是来自 localhost:3000 的前端我想用 POST 方法还要带 Content-Type 和 Authorization 这两个头你允不允许我这么干？
3. 服务器必须返回 CORS 允许头
plaintext
HTTP/1.1 200 OK
Access-Control-Allow-Origin: <http://localhost:3000>
Access-Control-Allow-Methods: POST,GET
Access-Control-Allow-Headers: Content-Type,Authorization
意思就是：
允许 localhost:3000 访问允许 POST、GET 方法允许你带 Content-Type、Authorization 这两个头
4. 浏览器校验结果
校验通过：才发送你真正的 POST 请求
校验不通过（没返回头 / 源不匹配 / 方法不允许）：
浏览器直接拦截真实请求，控制台报跨域错误，JS 完全拿不到任何响应。
👉 核心原因：安全

避免：

CSRF 风险扩大
非法跨域调用接口
敏感操作被滥用（DELETE / 修改数据）

👉 本质：

OPTIONS 是浏览器替你做的一次“权限确认”

在跨域场景下，浏览器为了安全，会对“复杂请求”先发送一个 OPTIONS 预检请求，询问服务器是否允许该跨域操作。
如果服务器返回允许的 CORS 头，浏览器才会发送真实请求，否则直接拦截。
本质上这是浏览器在同源策略下的一种安全校验机制。

# options 请求会携带 cookie 吗？

OPTIONS 预检请求，不会携带 Cookie。
再细一点说
不管你开没开 withCredentials: true前端就算配置了跨域带凭证：
js
axios.defaults.withCredentials = true
OPTIONS 里依然不带 Cookie
只有后面真正的 POST/GET/PUT 才会带 Cookie
浏览器就是这么设计的OPTIONS 只是用来问服务器：“我能不能跨域、能不能用这些头、能不能用这个方法？”它不代表真实业务请求，所以浏览器不会把身份信息（Cookie、Authorization）放进去。
后端也不需要在 OPTIONS 里读 Cookie后端处理 OPTIONS 一般直接放行，返回 CORS 头就行，不用鉴权。

# 简述浏览器的缓存策略 ？

它是为了在数据正确的情况下，减少网络请求，提升用户体验。

整个流程是：
当浏览器请求资源时，判断顺序是：

1️⃣ 先看 强缓存（本地直接用）
2️⃣ 强缓存失效 → 走 协商缓存（问服务器）
3️⃣ 协商失败 → 请求新资源

强缓存：
👉 特点：不会发请求，直接从浏览器拿

由服务器通过响应头控制：

1️⃣ Cache-Control（主流🔥）
Cache-Control: max-age=3600

👉 表示 1 小时内直接用缓存

常见值：

max-age=xxx：缓存时间
no-cache：可以缓存，但必须协商
no-store：完全不缓存

第一次请求资源时你第一次访问图片、JS、CSS 时，浏览器向服务器发请求，服务器把文件内容 + **缓存头（Cache-Control 等）** 一起返回给你。
浏览器自动存一份到本地浏览器看到缓存头，就把这个文件悄悄保存到本地：
快一点放 内存（Memory Cache）
持久一点放 硬盘（Disk Cache）
之后再访问这个资源时：
没过期 → 直接用这份本地存好的文件，不发请求
过期了 → 再去问服务器要不要更新
一句话：本地缓存 = 浏览器把你第一次加载的资源，偷偷备份在了自己电脑里。

三、协商缓存（发请求，但可能用缓存）

👉 特点：会发请求，但不一定下载资源

1️⃣ 基于时间：Last-Modified + If-Modified-Since

第一次响应：

Last-Modified: Wed, 21 Oct 2025 07:28:00 GMT

下一次请求：

If-Modified-Since: Wed, 21 Oct 2025 07:28:00 GMT

👉 服务器判断：

没变 → 304 Not Modified（用缓存）
变了 → 200 OK（返回新资源）
2️⃣ 基于标识：ETag（更精准🔥）

第一次响应：

ETag: "abc123"

下一次请求：

If-None-Match: "abc123"

👉 服务器判断：

一样 → 304
不一样 → 200

👉 缓存不仅影响性能，还影响前端发布策略（如版本更新、CDN、缓存穿透）

# 什么是图片防盗链？

简单说：防止别的网站直接 “盗用” 你服务器上的图片地址，白嫖你的服务器带宽和流量。

核心原理
浏览器在请求图片等资源时，HTTP 请求头会带上 Referer 字段，标识当前请求是从哪个页面发起的。
合法请求：Referer 是你自己的域名 → 正常返回图片
盗链请求：Referer 是别的网站域名 → 拒绝返回图片 / 返回默认 “防盗链图片”

1. Referer 校验防盗链（最经典、必考）
原理浏览器请求图片时会带上 Referer 请求头，表示当前页面来源。服务器判断来源域名是否合法，不合法就拒绝返回图片。
实现（Nginx）
nginx
location ~* \.(jpg|png|gif|webp)$ {
    valid_referers none blocked server_names *.your-site.com;
    if ($invalid_referer) {
        return 403;
    }
}
2. URL 签名 / Token 防盗链（更安全，大厂常用）
原理不给固定图片地址，而是生成带时间戳 + 签名的临时 URL，过期或签名不对就无法访问。
地址示例：/img/1.jpg?timestamp=xxx&sign=xxx
服务端校验 sign 和过期时间，防止永久盗用。

# 简述 HTTPS 加密过程

HTTPS = HTTP + TLS/SSL，核心是先用非对称加密安全交换对称密钥，再用对称加密高效传输数据，过程分为四步：
客户端发起握手客户端向服务端发送请求，携带支持的 TLS 版本、加密套件列表、随机数（Client Random）。
服务端响应并下发证书服务端选中加密套件，返回服务器随机数（Server Random），并发送数字证书（含服务器公钥、域名、CA 签名等）。
客户端验证证书并生成预主密钥客户端验证证书合法性（确认域名有效、CA 可信），生成预主密钥（Pre-Master Secret），用服务器公钥加密后发给服务端。
双方生成对称密钥并加密通信
客户端与服务端分别用 Client Random + Server Random + Pre-Master Secret，计算出相同的会话对称密钥；
后续所有 HTTP 数据，均用该对称密钥加密传输，既安全又高效。
简单总结：非对称加密传密钥，对称加密传内容。

# script 标签的 defer 和 async 有什么区别

一、先看：默认 <script> 的痛点
浏览器解析 HTML 时，遇到无属性的 <script> 会直接阻塞：
暂停解析 HTML
立即下载脚本
下载完成后立即执行
执行完毕，才继续解析 HTML
问题：脚本下载 / 执行期间，页面会白屏（渲染被阻塞），体验极差。

二、defer：延迟执行，顺序执行
核心特性
下载：异步下载脚本（不阻塞 HTML 解析）
执行时机：整个 HTML 文档解析完成后，DOMContentLoaded 事件之前执行
执行顺序：严格按照标签在页面中的书写顺序执行（多个defer脚本会排队）
作用范围：仅对外部脚本有效（必须带src，内联脚本无效）
代码示例
html
预览

```js
<!-- 两个脚本都会异步下载，但是 script1 一定先执行，script2 后执行 -->
<script src="script1.js" defer></script>
<script src="script2.js" defer></script>
```

三、async：异步执行，乱序执行
核心特性
下载：异步下载脚本（不阻塞 HTML 解析）
执行时机：脚本下载完成后立即执行（不管 HTML 有没有解析完）
执行顺序：谁先下载完，谁先执行（完全乱序，无依赖）
作用范围：仅对外部脚本有效（必须带src，内联脚本无效）
代码示例
html
预览

```js
<!-- 两个脚本异步下载，下载快的先执行，顺序不可控 -->
<script src="script1.js" async></script>
<script src="script2.js" async></script>
```

# WebSocket 和 HTTP 协议有什么区别

一句话总结：HTTP 是「一问一答」的单向短连接协议，WebSocket 是「双向实时通话」的持久长连接协议，WebSocket 本质是基于 HTTP 握手升级而来的全双工通信协议。

1. 通信模式：单向 vs 全双工
HTTP严格遵循请求 - 响应模型：只有客户端主动发起请求，服务端才能返回数据，服务端无法主动向客户端发消息。就像「发短信」，你发一条，对方回一条。
WebSocket建立连接后，客户端和服务端可以随时双向发送数据，无需等待请求。就像「打电话」，双方可以实时说话，无需轮流发起请求。

2. 连接方式：短连接 vs 持久长连接
HTTPHTTP/1.0 每次请求都新建 TCP 连接，请求完成立即断开；HTTP/1.1 支持 keep-alive 复用 TCP 连接，但本质仍是请求响应模式，连接空闲久了仍会断开，并非真正的长连接。
WebSocket一次 HTTP 握手成功后，协议升级为 WebSocket，TCP 连接会一直保持，直到客户端 / 服务端主动关闭，期间持续通信。

3.服务端推送能力
HTTP：无原生推送能力，只能通过轮询（定时发请求）、长轮询（请求挂起等待）模拟实时性，延迟高、浪费资源。
WebSocket：原生支持服务端主动推送消息，实时性毫秒级，资源占用极低。
4. 适用场景不同HTTP 适用于普通接口、页面资源等非实时场景；WebSocket 适用于聊天室、直播弹幕、行情数据、协同编辑等强实时需求。

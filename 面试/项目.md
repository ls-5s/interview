# 腾讯风格 · AI简历筛选系统 面试题（校招/社招全栈通用）
>
> 腾讯面试风格核心：**重基础、抠细节、强落地、追原理**，爱问“为什么这么设计”“有没有更好方案”“遇到坑怎么解决”。
> 以下题目严格贴合你简历技术栈，覆盖**一面基础、二面深度、三面架构/业务**，适配腾讯全栈/前端岗考察逻辑。

---

## 🎯 一面（基础关）：稳扎基础 + 项目落地能力

### 一、项目整体与基础认知（必问）

1. 介绍一下你的AI简历筛选SaaS项目，**用腾讯面试官的逻辑**：说清业务背景、解决的痛点、你负责的核心模块、以及效率提升的**具体数据**。
2. 项目技术栈是 `React + TypeScript + Vite + Node.js + Express + SQLite`，**腾讯风格追问**：
   - 为什么选 Vite 而不是 Webpack？在开发环境和生产环境分别做了哪些优化？
   - 为什么用 SQLite 而非 MySQL/PostgreSQL？SaaS场景下它的局限性是什么？后续怎么扩容？
3. 项目部署在 Vercel，**腾讯追问**：Vercel 的 Serverless 函数有什么限制？怎么解决冷启动问题？怎么保证接口的高可用？

### 二、前端基础与工程化（腾讯重点考察）

1. 你用 `React + Zustand` 做状态管理，**追问**：
   - Zustand 相比 Redux/Context 有什么优势？核心原理是什么（比如订阅发布模式、原子化更新）？
   - 项目中用 Zustand 管理了哪些核心状态？有没有遇到**状态同步/竞态**问题？怎么解决的？
2. TypeScript 在项目中的应用：**追问**
   - 怎么定义简历解析后的接口类型？有没有做**类型缩小/守卫**？遇到过联合类型/泛型的坑吗？
   - 怎么处理第三方库（如 pdf-parse）的类型声明？
3. 前端性能优化：**腾讯高频**
   - React Router 懒加载怎么实现？原理是什么？怎么配合 Vite 做代码分割（code splitting）？
   - 骨架屏（Skeleton）在项目中怎么落地？针对哪些场景做了适配？怎么保证骨架屏和真实内容的布局一致性？
   - 除了懒加载和骨架屏，还做了哪些前端性能优化？比如首屏加载、打包体积、渲染性能？
4. 简历解析与交互：
   - Mammoth / pdf-parse 解析 PDF/DOCX 时，怎么处理**格式错乱、乱码、换行符不一致**的问题？
   - 怎么提取姓名、邮箱、电话？除了正则，有没有用 LLM 做兜底解析？

### 三、后端与安全基础

1. JWT 双 Token 设计：**腾讯必问**
   - Access Token 和 Refresh Token 的过期时间怎么设计？Refresh Token 过期了怎么办？怎么实现自动续期？
   - 怎么防止 JWT 被劫持？HTTPS 之外，还做了哪些安全措施？
2. AES 加密 + 数据隔离：
   - AES 加密用在了哪些地方？（比如用户密码、简历敏感信息？）密钥怎么存储和管理？有没有做密钥轮换？
   - 多租户 SaaS 场景下，怎么实现**数据隔离**？保证不同 HR 看不到彼此的简历？
3. 接口设计与请求处理：
   - 基于 Axios 怎么封装请求？做了哪些拦截器？比如请求头、token 自动携带、响应统一处理、错误重试？

### 四、AI 基础认知

1. 七维度评分模型具体是哪七个维度？**追问**：设计依据是什么？每个维度的权重怎么分配？为什么这么分配？
2. Prompt 工程在项目中怎么落地？**追问**：用了哪些技巧（如 Few-Shot、Chain-of-Thought）？怎么让 LLM 输出更稳定、更符合要求？
3. LLM 输出不稳定问题，**三级解析机制（正则 + JSON + 兜底）** 详细说：
   - 每个层级的优先级是什么？适用场景是什么？怎么判断解析是否成功？怎么降低人工复核成本？

## 🎯 二面（深度关）：抠细节 + 问题解决能力

### 一、前端深度与工程化落地

1. 批量状态更新 + 邮件通知：**腾讯高频追细节**
   - 前端怎么处理批量操作的并发请求？比如批量筛选简历、批量更新状态？
   - 怎么管理 loading 状态？怎么处理请求失败/超时的重试逻辑？
   - 邮件通知是怎么集成的？前端怎么监听发送状态？做了哪些异常处理？
2. React 组件设计：
   - 怎么设计可复用的简历卡片、筛选组件？有没有做组件拆分、组合式 API？
   - 怎么处理组件的 props 透传、插槽？怎么保证组件的可维护性？
3. 多条件筛选优化：
   - 针对关键词/分数/时间多条件筛选，怎么做**防抖/节流**？怎么优化筛选性能？
   - 如果简历数据量达到 10 万+，前端怎么处理筛选？有没有做虚拟列表、分页加载？

### 二、后端深度与高可用

1. SQLite 并发与性能：
   - SQLite 在多用户并发访问时，怎么保证数据一致性？有没有做事务处理？
   - 后续用户量扩大，怎么从 SQLite 迁移到 MySQL/PostgreSQL？迁移过程中怎么保证数据不丢失？
2. 接口安全与防攻击：
   - 怎么防止 XSS、CSRF、SQL 注入攻击？在项目中具体做了哪些防护？
   - 怎么限制接口的访问频率（限流）？防止恶意请求？
3. 大模型 API 调用优化：
   - 怎么控制 LLM API 的调用成本？做了哪些缓存策略？
   - 怎么处理 API 调用失败、超时、限流？有没有做重试、降级方案？

### 三、AI 核心逻辑与难点

1. 七维度评分模型的优化：
   - 怎么验证模型的准确性？有没有做人工校验、A/B 测试？
   - 怎么根据不同岗位（前端/Java/产品）调整评分权重？怎么实现**岗位适配**？
2. LLM 输出稳定性优化：
   - 三级解析机制中，正则和 JSON 解析的优先级为什么这么定？遇到过模型输出格式错误怎么办？
   - 怎么让 LLM 输出可解释的评分理由？方便 HR 理解为什么匹配/不匹配？
3. 简历解析难点：
   - 怎么处理**非标格式简历**（比如手写转文字、图片简历）？目前项目有没有覆盖？后续怎么优化？
   - 怎么解析简历中的**项目经验、技能栈**？怎么避免技能提取错误？

## 🎯 三面（架构/业务关）：全局观 + 产品思维

### 一、架构设计（腾讯大厂必问）

1. 系统架构演进：
   - 目前的单体架构，后续用户量从 100 涨到 10 万，需要做哪些改造？比如拆分服务、引入微服务、消息队列？
   - 怎么设计**高可用、高并发**架构？怎么保证服务的容错性、可扩展性？
2. 多租户 SaaS 架构设计：
   - 完整的多租户设计方案是什么？包括租户管理、数据隔离、计费系统、权限系统？
   - 怎么保证 SaaS 系统的**安全性、合规性**（符合个人信息保护法）？
3. 集成与扩展：
   - 怎么把这个系统和企业现有 HR 系统（如 OA、招聘系统）集成？设计什么样的接口规范？
   - 后续怎么接入 AI 面试、候选人背调、JD 自动生成等功能？

### 二、业务与产品思维

1. 产品竞争力：
   - 对比市面上的 AI 简历筛选产品（如 Moka、BOSS 直聘 AI），你的项目有什么**差异化优势**？
   - 怎么让 AI 筛选更符合企业的**文化、价值观**？避免 AI 歧视？
2. 效率与成本平衡：
   - 怎么平衡 AI 筛选的效率和成本？比如控制 LLM API 调用次数、优化解析逻辑？
   - 效率提升 50%+ 是怎么量化的？有没有做数据对比？后续怎么进一步提升？
3. 产品迭代：
   - 项目后续的迭代规划是什么？怎么收集 HR 的反馈？怎么快速迭代功能？

### 三、复盘与优化

1. 项目中遇到的**最大技术难点**是什么？怎么解决的？如果重来，你会怎么优化？
2. 三级解析机制、七维度评分模型，有没有可以优化的空间？怎么提升模型的准确性和可解释性？
3. 项目的**性能瓶颈**在哪里？怎么优化？比如数据库查询、接口响应、前端渲染？

## 🎯 腾讯高频「连环追问」（必准备）
>
> 腾讯面试官喜欢连环追，以下是结合你项目的**必问追题**，准备好就能稳过。

1. 你说用 Zustand 做状态管理，**追问**：Zustand 的中间件怎么用？你项目中有没有自定义中间件？
2. 三级解析机制，**追问**：如果正则解析失败，LLM 兜底的 Prompt 是怎么写的？怎么保证 LLM 输出符合 JSON 格式？
3. JWT 双 Token，**追问**：怎么实现 Token 自动续期？前端怎么监听 Token 过期并跳转登录页？
4. 简历解析，**追问**：怎么处理简历中的**特殊字符、表情、表格**？解析准确率目前是多少？
5. 性能优化，**追问**：Vite 打包时怎么优化？怎么分析打包体积？有没有做 Tree Shaking？
6. 安全，**追问**：AES 加密的密钥是存在环境变量还是配置中心？怎么防止密钥泄露？

### 💡 腾讯面试准备技巧

1. **答原理不答表面**：比如问 React 懒加载，不仅要说实现，还要说清 `React.lazy + Suspense` 原理、Vite 如何处理代码分割。
2. **带数据说话**：效率提升 50%+、解析准确率多少、打包体积优化多少，用数据证明你的工作。
3. **主动讲解决问题的过程**：腾讯更看重你“遇到坑怎么解决”，而不是你“做了多少功能”。
4. **结合腾讯业务场景**：比如多租户、高可用、安全、SaaS 化，这些都是腾讯内部系统的核心需求，多往这靠。

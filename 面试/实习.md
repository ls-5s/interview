# 为什么长列表会卡顿？

长列表卡顿，核心是一次性渲染大量 DOM 节点，占用浏览器主线程；滚动时触发频繁重排重绘，主线程阻塞导致页面掉帧、交互卡顿。
标准精简回答（面试官最爱）
一次性渲染海量 DOM，初始化渲染与内存占用过高；
滚动过程中频繁触发重排（Reflow）、重绘（Paint），性能开销大；
JS 执行、DOM 操作与渲染共用主线程，大量计算阻塞交互响应。

# 虚拟滚动的核心原理是什么？

假设场景：总数据 10000 条，每条高度固定 50px，可视区域高度 500px，最多显示 10 条。
设置虚拟总高度，撑开滚动条用「总条数 × 单条高度」计算出完整列表高度，让滚动条长度和真实列表一致。样例：10000 × 50px = 500000px，给容器设置这个高度占位，滚动行为和真实长列表一样。
监听滚动位置，计算可视区起始索引滚动时通过 scrollTop 算出当前从第几条开始显示。样例：滚动距离 scrollTop = 500px，500 ÷ 50 = 10，即从第 10 条开始显示。
只渲染可视区域 + 少量缓冲数据不渲染全部 10000 条，只渲染当前可见的 10 条，再加前后几条缓冲防止白屏。样例：从第 10 条开始，只渲染 10~25 条，DOM 数量始终保持十几条，不随数据量增加。
通过偏移定位，让内容出现在正确位置用 transform 或 top 把渲染的内容向下偏移，对齐滚动位置，视觉上无缝滚动。样例：从第 10 条开始，就把内容区域偏移 10×50=500px，看起来就像滚动到了对应位置。
滚动时复用 / 更新 DOM，不重建大量节点滚动后只更新内容和偏移量，不新增删除 DOM，减少重排重绘。样例：滚动到第 20 条时，直接替换原有 DOM 里的内容，再修改偏移为 1000px，不新建 DOM。

# 你是怎么实现虚拟滚动的？用了什么库还是自己写的？

定义容器：设置固定可视区域高度，监听容器滚动事件；
计算核心参数：根据滚动偏移量，计算可视区起始 / 结束索引，搭配少量缓冲条目防止白屏；
虚拟高度撑开：用总数据量 × 单条高度，设置容器总高度，模拟真实滚动条；
DOM 渲染优化：只渲染可视 + 缓冲区域的节点，滚动时复用现有 DOM，仅更新内容和偏移位置，不大量增删 DOM；
兼容优化：缓存条目高度，适配固定高度场景，滚动事件做节流，降低计算开销。
深挖题（腾讯最爱问）

```js
// 1. 基础配置（固定参数）
const itemHeight = 50;    // 每一行的高度 50px
const bufferSize = 5;     // 上下缓冲5条，防止滚动白屏
const viewHeight = 500;   // 可视区域高度 500px
const totalList = Array.from({ length: 10000 }, (_, i) => `数据 ${i}`); // 模拟1万条数据

// 2. 获取页面元素
const container = document.querySelector('.virtual-container'); // 滚动容器
const scrollSizer = document.querySelector('.scroll-sizer');    // 撑开滚动条的元素
const listRender = document.querySelector('.list-render');      // 渲染列表内容的元素

// 3. 设置虚拟总高度（撑开滚动条，模拟完整列表）
scrollSizer.style.height = totalList.length * itemHeight + 'px';

// 4. 节流函数（优化性能：滚动时减少计算次数，不卡顿）
const throttle = (fn) => {
  let timer = null;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), 10);
  };
};

// 5. 核心渲染函数（虚拟滚动的关键）
const render = () => {
  // 获取滚动距离
  const scrollTop = container.scrollTop;
  
  // 计算当前可视区第一条数据的索引
  const start = Math.floor(scrollTop / itemHeight);
  
  // 计算带缓冲的渲染范围（上下多渲染5条）
  const realStart = Math.max(0, start - bufferSize);
  const realEnd = start + Math.floor(viewHeight / itemHeight) + bufferSize;
  
  // 截取需要渲染的数据（只渲染可视+缓冲，不渲染1万条）
  const renderData = totalList.slice(realStart, realEnd);
  
  // 偏移定位：把内容挪到正确位置，模拟滚动效果
  listRender.style.transform = `translateY(${realStart * itemHeight}px)`;
  
  // 渲染DOM：只更新内容，复用固定数量的节点
  listRender.innerHTML = renderData.map(item => `<div class="item">${item}</div>`).join('');
};

// 6. 监听容器滚动，触发渲染
container.addEventListener('scroll', throttle(render));

// 7. 页面初始化，第一次渲染
render();
```

用虚拟高度撑开滚动条，只渲染可视区 + 缓冲的 DOM，滚动时通过计算偏移量复用节点、更新内容，从根源减少 DOM 操作，保证长列表流畅不卡顿。

# 虚拟滚动中，如何处理高度不固定的列表项？

动态高度虚拟滚动的核心是：缓存所有列表项的真实渲染高度 + 建立位置索引映射，通过二分查找快速定位滚动位置，替代固定高度的直接计算。

高度缓存
初始化 / 首次渲染时，获取每个列表项的真实 DOM 高度，存入数组永久缓存；同时计算并缓存累计偏移量（前 n 项高度总和），建立索引与位置的映射关系。
二分查找定位
滚动时，不再用 scrollTop / 固定行高 计算索引，而是通过二分查找滚动距离，在累计偏移量数组中快速匹配当前可视区的起始 / 结束索引。
常规渲染逻辑
依旧只渲染可视区 + 缓冲 DOM，复用节点；偏移量直接读取缓存的累计高度，保证位置精准。
兼容更新
列表数据变更、容器 resize 时，重新采集对应项的高度并更新缓存，保证数据同步。

# 快速滚动时出现白屏怎么办？你是怎么优化的？

1. 增加上下缓冲区域（最核心、最有效）
在可视区域上下额外多渲染几条数据（缓冲层），快速滚动时，缓冲内容提前加载，永远不会出现空白区域。
对应之前的 bufferSize 配置，一般设 5~10 条。
2. 滚动事件做节流优化
限制滚动事件的触发频率，避免高频触发渲染函数，减少浏览器计算压力，防止卡顿导致白屏。
3. 开启 CSS 硬件加速
用 transform: translateY() 设置偏移，触发浏览器GPU 硬件加速，避免频繁回流重绘，提升渲染速度。
4. 极致复用 DOM 节点
不销毁、不新建 DOM，只替换节点内的文字 / 内容，把 DOM 操作降到最低，大幅提升渲染效率。
5. 预渲染首屏 / 边缘内容
初始化时提前渲染缓冲数据，滚动到列表顶部 / 底部时，提前加载边缘内容，杜绝边缘白屏。

# 为什么数组查找的时间复杂度是 O (n)，Map 是 O (1)？

1. 数组查找为什么是 O (n)？
数组的按值查找，需要从头挨个遍历对比，直到找到目标元素。
最坏情况要找遍整个数组的 n 个元素
所以时间复杂度是 O(n)
注意：数组按索引访问是 O (1)，但按值查找是 O (n)。
2. Map（哈希表）为什么是 O (1)？
Map 底层是哈希表，通过哈希函数把 key 直接计算出对应的内存地址，一步定位，不需要遍历。
不管数据多少，查找都只做一次计算
所以时间复杂度是 O(1)

# 你是在什么场景下做的这个优化？

在我开发的 IMA Studio 影视协作平台 里，针对虚拟滚动长列表的高频权限校验场景做了优化。滚动时数组遍历查找权限效率低，我用 Map 缓存替代数组遍历，将时间复杂度从 O (n) 降到 O (1)，解决了滚动卡顿和白屏问题。

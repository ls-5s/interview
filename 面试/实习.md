基础题

# 为什么长列表会卡顿？

长列表卡顿，核心是一次性渲染大量 DOM 节点，占用浏览器主线程；滚动时触发频繁重排重绘，主线程阻塞导致页面掉帧、交互卡顿。
标准精简回答（面试官最爱）
一次性渲染海量 DOM，初始化渲染与内存占用过高；
滚动过程中频繁触发重排（Reflow）、重绘（Paint），性能开销大；
JS 执行、DOM 操作与渲染共用主线程，大量计算阻塞交互响应。

# 虚拟滚动的核心原理是什么？

虚拟滚动的核心是以 “计算驱动视图” 替代 “DOM 驱动视图”，只渲染可视区域内的 DOM 节点，通过动态复用 / 替换节点、同步数据位置，彻底减少 DOM 数量与重排重绘开销，实现长列表流畅交互。
核心逻辑拆解（面试分点口述版）
可视区域聚焦不一次性渲染所有列表项，仅计算当前屏幕可见区域的节点范围，只生成对应 DOM，初始 DOM 数量从 “百 / 千级” 降至 “10-30 级”，从根源降低渲染压力。
节点复用与动态替换滚动时不频繁创建 / 销毁 DOM，而是通过移动现有节点位置+更新节点对应数据的方式，实现列表项的 “视觉滚动”；同时预留少量缓冲区域，提前渲染即将进入可视区的节点，避免滚动空白。
位置同步与滚动联动维护一个滚动容器的虚拟高度（由所有列表项高度总和计算），模拟列表整体高度；滚动时实时计算可视区起始索引，同步更新 DOM 的位置和内容，让浏览器感知不到 “DOM 变少”，视觉上与普通长列表无差异。

# 你是怎么实现虚拟滚动的？用了什么库还是自己写的？

定义容器：设置固定可视区域高度，监听容器滚动事件；
计算核心参数：根据滚动偏移量，计算可视区起始 / 结束索引，搭配少量缓冲条目防止白屏；
虚拟高度撑开：用总数据量 × 单条高度，设置容器总高度，模拟真实滚动条；
DOM 渲染优化：只渲染可视 + 缓冲区域的节点，滚动时复用现有 DOM，仅更新内容和偏移位置，不大量增删 DOM；
兼容优化：缓存条目高度，适配固定高度场景，滚动事件做节流，降低计算开销。

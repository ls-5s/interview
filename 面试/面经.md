# 为什么在本地使用ai 要使用sse ,而不是wesokcet,怎么去实现打字机的效果

ai 的输出的单项
用户提问 -> 服务器推荐 -> token 一点点放回 -> 前端展示
👉 关键点：

客户端不需要持续发消息
只需要“接收流”

所以：

技术 是否匹配 AI 场景
SSE ✅ 完全匹配（单向流）
WebSocket ❌ 过重（双向通信）

方式二：前端再做“慢一点”打字机

👉 如果后端返回太快，可以前端控制节奏：

let queue = []
let showing = ''

es.onmessage = (e) => {
  queue.push(e.data)
}

setInterval(() => {
  if (queue.length > 0) {
    showing += queue.shift()
    document.getElementById('app').innerText = showing
  }
}, 30)

👉 作用：

控制“打字速度”
更丝滑
为什么不用 WebSocket 做打字机？

👉 你可以这样答：

WebSocket 适合双向通信，但 AI 场景是服务端持续推送数据，用 SSE 更轻量；打字机效果本质是前端控制渲染节奏，与通信协议无关。

# 9了解RAG吗（粗略的说了一下）

👉 RAG（Retrieval-Augmented Generation）是一种“先检索资料，再让大模型生成答案”的技术，用来提升回答的准确性和时效性。

用户提问
   ↓
去知识库查资料（检索）
   ↓
把资料 + 问题一起给大模型
   ↓
生成答案

【5️⃣ 和微调的区别（经典面试题🔥）】

👉 一句话对比：

RAG：查资料再答（开卷）
微调：把知识训练进模型（背书）

RAG 是一种检索增强生成技术，在用户提问时先从外部知识库检索相关信息，再将检索结果作为上下文输入给大模型生成答案。这样可以有效降低模型幻觉，同时支持私有数据和实时更新，是当前大模型落地的核心方案之一。
